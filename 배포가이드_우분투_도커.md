## 0. 전체 그림

```
[사용자 브라우저]
      │ https://consol.sae-a.com (443)
      ▼
[nginx]  ─ SSL 종료(Let's Encrypt), 80→443 리다이렉트
      │ proxy_pass → 127.0.0.1:5000
      ▼
[Docker: consol-web 컨테이너]  ─ gunicorn(워커1·스레드16) + Flask app.py
      │ 바인드 마운트
      ▼
[/srv/consol-app]  ─ 코드 + 데이터(uploads/, results/, *.json, secret_key.bin)

[GitHub/GitLab 저장소] ──(push 감지)──▶ [Jenkins :8080]
                                            │ ① 문법검사 ② 코드만 rsync
                                            │ ③ docker compose 재시작 ④ 헬스체크
                                            ▼
                                       /srv/consol-app
```

### 코드 vs 데이터 구분 (CI/CD의 핵심 — 코드로 전수 확인한 사실)

이 앱은 DB 없이 모든 상태를 프로젝트 루트의 파일로 관리합니다. **git이 추적하는 것은 코드뿐**이고, 서버에서 변하는 데이터는 Jenkins 배포가 절대 건드리지 않습니다.

| 구분 | 항목 | 배포 방법 |
|---|---|---|
| **코드 (git 추적)** | `*.py`, `templates/`, `requirements.txt`, `Dockerfile`, `docker-compose.yml`, `Jenkinsfile`, `consol_template.json`(읽기전용 확인됨) | git push → Jenkins 자동 배포 |
| **데이터 (git 제외, 서버에만 존재)** | `uploads/`(패키지·`_state.json`·journals), `results/`, `secret_key.bin`, `auth_config.json`, `permission_groups.json`, `years_config.json`, `wce_overrides.json`, `fx_rates.json`, `cash_worksheet_manuals.json`, `prior_consolidated.json`, `consol_groups.json`, `consol_journals.json`, `distribute_passwords.json`, `cf_mapping_v2_draft.json`, `distribute_template/*.xlsm`(관리자 UI로 업로드됨) | 최초 1회 전송 후 서버에서만 관리, 백업으로 보호 |
| **데이터 (읽기전용, 수동 교체)** | `필수제출회사리스트.xlsx` | 변경 시 서버에 scp로 직접 교체 |

위 데이터 목록은 `app.py`/`consol_schema.py`/`distribute_builder.py`의 저장 함수(`_save_credentials`, `save_journals`, `save_template` 등)를 전부 추적해 확인한 것입니다. `.gitignore`에 이미 반영되어 있습니다.

### 운영 구조상 주의 2가지 (왜 이렇게 설계했는가)

1. **gunicorn 워커는 반드시 1개** — 업로드 목록이 프로세스 메모리 전역변수에 캐시되고 백그라운드 스레드가 `uploads/_state.json`으로 비동기 저장하는 구조(`app.py:7203`). 워커 2개 이상이면 워커 간 목록이 어긋남. 동시 사용자는 스레드 수(`--threads`)로 감당.
2. **이미지에 코드를 굽지 않고 디렉터리 전체를 바인드 마운트** — 코드와 데이터가 한 디렉터리에 섞여 있는 현재 구조에서 가장 안전. 배포 = "추적 파일만 rsync + 컨테이너 재시작"이라 이미지 재빌드는 `requirements.txt` 변경 시에만 발생.

### 사전 준비물 체크리스트

- [ ] Ubuntu 22.04/24.04 LTS 서버 (sudo 가능 계정)
- [ ] DNS: `consol.sae-a.com` A 레코드 → 서버 IP (사내 DNS 또는 공개 DNS 관리자에게 요청)
- [ ] Let's Encrypt 사용 시: 인터넷에서 서버의 80/443 포트 접근 가능해야 함 (불가하면 6.4절의 대안)
- [ ] Git 원격 저장소 1개 (GitHub private 또는 사내 GitLab) — 빈 저장소 `consol-app` 생성
- [ ] 방화벽/보안그룹: 22(SSH), 80, 443 오픈. 8080(Jenkins)은 사내망만

## 3. 운영 디렉터리 구성 (최초 1회 — 데이터 포함 전체 이전)

git에는 코드만 있으므로, **최초 1회는 데이터를 포함한 전체 디렉터리를 직접 전송**합니다.

### 3.1 전송

**주의: Windows 탐색기 zip 압축 금지** — 한글 파일명이 CP949로 깨집니다. scp를 쓰세요 (Windows 11 기본 내장):

```powershell
scp -r "C:\패키지프로그램" 사용자명@서버IP:/tmp/consol-app
```

서버에서 운영 위치로 이동 + Windows 전용 항목 제거:

```bash
sudo mv /tmp/consol-app /srv/consol-app
cd /srv/consol-app
sudo rm -rf python/ venv/ __pycache__/ _slide_preview/ ppt_screenshots/
sudo rm -f setup_and_run.bat server.log _smoke_server.log
```

### 3.2 경로 마이그레이션 (필수 — 건너뛰면 기존 업로드 목록 전체 소실)

`uploads/_state.json`에 업로드 경로가 Windows 백슬래시(`"uploads\\16ddc914_..."`)로 저장되어 있습니다. 시작 시 `_load_state()`가 "실존 파일만" 남기는데 리눅스에서 백슬래시 경로는 없는 파일로 판정되어 목록이 전부 제거됩니다. 최초 1회 변환:

```bash
cd /srv/consol-app
sudo python3 - <<'EOF'
import json
p = 'uploads/_state.json'
data = json.load(open(p, encoding='utf-8'))
for e in data:
    if 'path' in e:
        e['path'] = e['path'].replace('\\', '/')
json.dump(data, open(p, 'w', encoding='utf-8'), ensure_ascii=False, indent=2)
print(f'{len(data)}건 경로 변환 완료')
EOF
```

(루트의 다른 JSON에는 백슬래시 경로가 없음을 확인했습니다. 이후 생성되는 경로는 리눅스 형식이므로 1회면 충분합니다.)

## 5. nginx 설치

```bash
sudo apt -y install nginx
```

`/etc/nginx/sites-available/consol` 생성:

```nginx
server {
    listen 80;
    server_name consol.sae-a.com;

    # 앱 업로드 한도가 50MB(MAX_CONTENT_LENGTH)이므로 그보다 크게
    client_max_body_size 60m;
    # 패키지 일괄 검증 등 장시간 요청 대비 (gunicorn timeout 600과 맞춤)
    proxy_read_timeout 600s;
    proxy_send_timeout 600s;

    location / {
        proxy_pass http://127.0.0.1:5000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

활성화:

```bash
sudo ln -s /etc/nginx/sites-available/consol /etc/nginx/sites-enabled/
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t && sudo systemctl reload nginx
```

DNS가 잡혀 있다면 이 시점에 `http://consol.sae-a.com` 접속이 되어야 합니다(아직 HTTP).

## 6. 도메인 + SSL (HTTPS)

### 6.1 DNS 확인

```bash
dig +short consol.sae-a.com    # 서버 IP가 나와야 함
```

안 나오면 DNS 관리자에게 A 레코드 등록을 요청하고 전파를 기다립니다.

### 6.2 Let's Encrypt 인증서 발급 (서버가 인터넷에서 80포트로 접근 가능한 경우)

```bash
sudo apt -y install certbot python3-certbot-nginx
sudo certbot --nginx -d consol.sae-a.com
# 이메일 입력 → 약관 동의 → "Redirect" 선택 (HTTP→HTTPS 자동 리다이렉트)
```

certbot이 nginx 설정에 443 블록과 인증서 경로를 자동 추가합니다. 확인:

```bash
sudo nginx -t && sudo systemctl reload nginx
curl -I https://consol.sae-a.com/
```

**자동 갱신**: certbot 패키지가 systemd 타이머로 자동 갱신합니다. 확인만 해두세요:

```bash
systemctl list-timers | grep certbot
sudo certbot renew --dry-run    # 갱신 리허설 성공 확인
```

### 6.3 발급 후 nginx 최종 형태 (참고)

certbot 적용 후 `/etc/nginx/sites-available/consol`은 대략 이렇게 됩니다 — `client_max_body_size`/`proxy_read_timeout`이 **443 블록 안에 남아있는지** 확인하세요:

```nginx
server {
    server_name consol.sae-a.com;
    client_max_body_size 60m;
    proxy_read_timeout 600s;
    proxy_send_timeout 600s;

    location / {
        proxy_pass http://127.0.0.1:5000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    listen 443 ssl;
    ssl_certificate /etc/letsencrypt/live/consol.sae-a.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/consol.sae-a.com/privkey.pem;
    include /etc/letsencrypt/options-ssl-nginx.conf;
    ssl_dhparam /etc/letsencrypt/ssl-dhparams.pem;
}
server {
    if ($host = consol.sae-a.com) { return 301 https://$host$request_uri; }
    listen 80;
    server_name consol.sae-a.com;
    return 404;
}
```

### 6.4 서버가 사내망 전용인 경우 (인터넷에서 80포트 접근 불가)

HTTP-01 방식의 certbot은 쓸 수 없습니다. 둘 중 하나:

- **DNS-01 챌린지**: `sae-a.com` DNS가 Route53/Cloudflare 등 API 지원 DNS면 `certbot-dns-*` 플러그인으로 발급 가능. DNS 관리자와 협의 필요.
- **회사 인증서**: IT부서가 보유한 `*.sae-a.com` 와일드카드(또는 사내 CA) 인증서를 받아 직접 설치:
  ```nginx
  listen 443 ssl;
  ssl_certificate     /etc/nginx/ssl/consol.sae-a.com.crt;   # 체인 포함 fullchain
  ssl_certificate_key /etc/nginx/ssl/consol.sae-a.com.key;
  ```
  이 경우 만료 관리(보통 1년)는 IT부서 일정에 맞춰 수동 교체합니다.

## 7. Jenkins 설치 및 CI/CD 파이프라인

### 7.1 Jenkins 설치 (운영 서버에 직접 설치)

```bash
# Java 17
sudo apt -y install openjdk-17-jre

# Jenkins LTS 저장소 등록 및 설치

sudo apt update && sudo apt -y install jenkins

sudo systemctl enable --now jenkins
systemctl status jenkins
```

### 7.2 Jenkins 권한 설정 (중요)

jenkins 계정이 ① docker를 다루고 ② 운영 디렉터리에 코드를 쓸 수 있어야 합니다:

```bash
sudo usermod -aG docker jenkins
sudo chown -R jenkins:jenkins /srv/consol-app
sudo systemctl restart jenkins
```

### 7.3 초기 설정

브라우저에서 `http://서버IP:8080` 접속 (사내망에서만 — 9장 방화벽 참조):

1. 초기 비밀번호 입력: `sudo cat /var/lib/jenkins/secrets/initialAdminPassword`
2. **Install suggested plugins** 선택 (Git, Pipeline 포함됨)
3. 관리자 계정 생성

### 7.4 Git 저장소 접근 자격증명

GitHub private 저장소 기준:

1. GitHub에서 **Deploy Key** 발급이 깔끔합니다. 서버에서:
   ```bash
   sudo -u jenkins ssh-keygen -t ed25519 -f /var/lib/jenkins/.ssh/id_ed25519 -N ""
   sudo -u jenkins cat /var/lib/jenkins/.ssh/id_ed25519.pub
   ```
2. 출력된 공개키를 GitHub 저장소 → Settings → Deploy keys에 등록 (read-only면 충분)
3. Jenkins → Manage Jenkins → Credentials → Add: **SSH Username with private key**, username `git`, 개인키(`id_ed25519` 내용) 붙여넣기
4. 최초 1회 호스트키 등록:
   ```bash
   sudo -u jenkins ssh -T git@github.com    # yes 입력 (인증 성공 메시지 또는 "does not provide shell access"면 정상)
   ```

(사내 GitLab도 동일 — Deploy Key 또는 Access Token 방식)

### 7.5 파이프라인 Job 생성

저장소에 이미 `Jenkinsfile`이 포함되어 있습니다 (체크아웃 → 문법검사 → 코드 rsync → 컨테이너 재시작 → 헬스체크).

1. Jenkins 첫 화면 → **New Item** → 이름 `consol-app-deploy` → **Pipeline** 선택
2. Pipeline 섹션:
   - Definition: **Pipeline script from SCM**
   - SCM: **Git**, Repository URL: `git@github.com:회사조직/consol-app.git`
   - Credentials: 7.4에서 등록한 키 선택
   - Branch: `*/main`
   - Script Path: `Jenkinsfile` (기본값)
3. 저장 → **Build Now**로 첫 배포 실행 → 콘솔 출력에서 5개 stage 모두 통과 확인

### 7.6 자동 트리거

`Jenkinsfile`에 `pollSCM('H/2 * * * *')`이 들어 있어 **2분 간격으로 git 변경을 감지해 자동 배포**합니다. 별도 설정 없이 동작합니다 (단, Job을 최초 1회는 수동 빌드해야 트리거가 등록됩니다).

서버가 인터넷에서 접근 가능하고 즉시 배포를 원하면 webhook으로 전환할 수 있습니다: GitHub 저장소 → Settings → Webhooks → `https://서버주소:8080/github-webhook/` 등록 + Jenkinsfile에서 `triggers` 블록 제거 + Job 설정에서 "GitHub hook trigger" 체크. 사내망 전용 서버라면 pollSCM을 그대로 쓰는 것이 간단합니다.

### 7.7 이후의 배포 플로우 (개발자 일상)

```powershell
# Windows 개발 PC에서
cd C:\패키지프로그램
git add app.py templates/xxx.html      # 수정한 코드 파일
git commit -m "패키지 검증 항목 추가"
git push
# → 2분 내 Jenkins가 감지하여 자동 배포 (재시작 수 초 단절 발생)
```

> 배포 시 컨테이너가 재시작되므로 **수 초간 접속이 끊깁니다**. 사용자가 많은 시간대(결산 마감 등)에는 push를 피하거나, push만 해두고 Jenkins Job을 일시 중지(Disable)했다가 업무 후 빌드하세요.

## 8. 일상 운영

### 8.1 명령 모음 (`/srv/consol-app`에서)

| 작업 | 명령 |
|---|---|
| 상태 확인 | `docker compose ps` |
| 앱 로그 (실시간) | `docker compose logs -f --tail 200` |
| 수동 재시작 | `docker compose restart` |
| 중지 / 시작 | `docker compose down` / `docker compose up -d` |
| nginx 로그 | `sudo tail -f /var/log/nginx/access.log` |
| Jenkins 로그 | Jenkins 웹 UI → 해당 빌드 → Console Output |

- 서버 재부팅 시 docker(`restart: unless-stopped`), nginx, jenkins 모두 자동 시작됩니다.
- **`docker kill` 금지** — 종료 시 atexit가 업로드 목록을 `_state.json`에 flush하므로 항상 `docker compose down/restart`를 쓰세요 (`stop_grace_period: 60s` 설정됨).

### 8.2 필수제출회사리스트 교체 (분기마다)

git에 없는 파일이므로 직접 교체합니다:

```powershell
scp "필수제출회사리스트.xlsx" 사용자명@서버IP:/tmp/
```
```bash
sudo mv /tmp/필수제출회사리스트.xlsx /srv/consol-app/
sudo chown jenkins:jenkins /srv/consol-app/필수제출회사리스트.xlsx
# 재시작 불필요 — 요청 때마다 파일을 새로 읽음
```

### 8.3 백업 (매일 새벽 3시, 14일 보관)

모든 상태가 `/srv/consol-app` 안의 파일이므로 디렉터리 백업이 곧 전체 백업입니다:

```bash
sudo mkdir -p /srv/backups
sudo crontab -e
# 추가:
0 3 * * * tar -czf /srv/backups/consol-$(date +\%Y\%m\%d).tar.gz -C /srv consol-app && find /srv/backups -name 'consol-*.tar.gz' -mtime +14 -delete
```

가능하면 백업 파일을 다른 장비(NAS 등)로도 복제하세요. 서버 디스크가 죽으면 같은 디스크의 백업도 같이 죽습니다.

### 8.4 복구 절차

```bash
cd /srv/consol-app && docker compose down
sudo tar -xzf /srv/backups/consol-YYYYMMDD.tar.gz -C /srv   # consol-app 디렉터리 복원
sudo chown -R jenkins:jenkins /srv/consol-app
cd /srv/consol-app && docker compose up -d
```

## 9. 보안

### 9.1 방화벽 (ufw)

```bash
sudo ufw allow OpenSSH
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw allow from 사내망대역/24 to any port 8080 proto tcp   # Jenkins는 사내망만 (예: 192.168.10.0/24)
sudo ufw enable
sudo ufw status
```

- 앱 포트 5000은 `127.0.0.1` 바인딩이라 방화벽과 무관하게 외부 접근이 불가합니다.
- 참고: 도커가 직접 publish하는 포트는 ufw를 우회할 수 있으나, 이 구성은 localhost 바인딩이므로 해당 없음.

### 9.2 파일 권한 (3.3에서 수행했는지 확인)

```bash
ls -l /srv/consol-app/secret_key.bin /srv/consol-app/auth_config.json /srv/consol-app/distribute_passwords.json
# -rw------- (600) 이어야 함
```

### 9.3 기타 권장

- SSH 비밀번호 로그인 비활성화(키 인증만): `/etc/ssh/sshd_config`에서 `PasswordAuthentication no`
- Jenkins 관리자 외 계정 생성 금지, "Anyone can do anything" 권한 설정 금지 (기본값 유지)
- `sudo apt -y install fail2ban` — SSH 무차별 대입 차단 (기본 설정으로 충분)
- git 저장소는 반드시 **private** — 코드 자체에 내부 계정 구조·검증 로직이 들어 있음

## 10. 문제 해결

| 증상 | 확인/조치 |
|---|---|
| 컨테이너가 곧바로 죽음 | `docker compose logs` — import 오류면 파일 누락. Jenkins 배포 직후라면 해당 빌드의 Syntax Check 통과 여부 확인 |
| 업로드 목록이 비어 보임 | 3.2 경로 마이그레이션 수행 여부, `uploads/` 전송 여부 |
| 한글 파일명 깨짐 | zip으로 전송한 경우 → scp로 재전송 |
| https 접속 불가, http만 됨 | `sudo nginx -t`, certbot 발급 성공 여부(`sudo certbot certificates`) |
| 인증서 만료 경고 | `sudo certbot renew --dry-run`으로 갱신 점검, `systemctl list-timers \| grep certbot` |
| 50MB 초과 업로드 실패 | 앱 자체 한도(정상). nginx `client_max_body_size 60m`이 443 블록에 있는지 확인 |
| 일괄 검증 중 504 | nginx `proxy_read_timeout 600s` + gunicorn `--timeout 600` 확인 |
| TOTP 로그인 일괄 실패 | 서버 시계 (`timedatectl`) |
| Jenkins 빌드 실패: docker 권한 | `sudo usermod -aG docker jenkins && sudo systemctl restart jenkins` |
| Jenkins 빌드 실패: rsync Permission denied | `sudo chown -R jenkins:jenkins /srv/consol-app` |
| Jenkins가 push를 감지 안 함 | Job을 최초 1회 수동 빌드했는지(pollSCM 등록 조건), Git 자격증명 유효성 |
| push했는데 코드가 반영 안 됨 | 해당 파일이 `.gitignore`에 걸려 추적 제외됐는지 `git status`로 확인 |
