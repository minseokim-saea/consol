# 연결 재무보고 통합 시스템 — 우분투 서버 도커 배포 가이드

작성일: 2026-06-12

## 0. 시스템 구조 요약 (배포 관점)

코드를 직접 확인하여 정리한 사실입니다.

- **앱**: Flask 3.0.3 단일 애플리케이션(`app.py`), 포트 **5000**, `host 0.0.0.0`
- **의존성**: `requirements.txt`의 8개 패키지(flask, openpyxl, pandas, xlrd, xlwt, werkzeug, filelock, msoffcrypto-tool) — 전부 순수 파이썬 계열이라 리눅스에서 그대로 동작 (Windows 전용 모듈 사용 없음, 코드 grep으로 확인)
- **데이터 저장 위치**: 전부 **프로젝트 루트 기준 상대경로** (DB 없음, 파일 기반)
  - `uploads/` — 업로드된 패키지 원본 + `_state.json`(업로드 목록) + `journals/`(분개 원본)
  - `results/` — 산출물
  - 루트의 mutable JSON: `auth_config.json`(계정), `years_config.json`, `wce_overrides.json`, `fx_rates.json`, `consol_journals.json`, `consol_groups.json`, `permission_groups.json`, `distribute_passwords.json`, `prior_consolidated.json`, `cf_mapping_v2_draft.json` 등
  - `secret_key.bin` — Flask 세션 키 (없으면 자동 생성; 바뀌면 전 사용자 재로그인)
  - 런타임에 읽는 고정 파일: `필수제출회사리스트.xlsx`, `distribute_template/2026.xlsm`, `cash_worksheet_manuals.json`, `consol_template.json`
- **동시성 구조 (중요)**: 업로드 목록(`uploaded_files`)이 **프로세스 메모리의 전역 변수**에 캐시되고, 백그라운드 스레드가 `uploads/_state.json`으로 비동기 flush합니다(`app.py`의 `_save_state`/`_load_state`). 따라서 **웹서버 프로세스(워커)는 반드시 1개**여야 합니다. 워커를 2개 이상 띄우면 워커끼리 업로드 목록이 어긋납니다. 동시 접속 처리는 스레드로 해결합니다(원래 코드도 `threaded=True` 설계).

이 가이드의 배포 방식: **이미지에는 파이썬+의존성만 굽고, 프로젝트 디렉터리 전체를 컨테이너에 바인드 마운트**합니다. 코드와 데이터가 한 디렉터리에 섞여 있는 현재 구조에서 가장 단순하고 안전한 방식이며, 코드 수정 시 이미지 재빌드 없이 파일 교체 + 컨테이너 재시작만 하면 됩니다.

준비된 파일 3개 (이 저장소에 포함):

| 파일 | 역할 |
|---|---|
| `Dockerfile` | python:3.12-slim + requirements + gunicorn |
| `docker-compose.yml` | 포트 5000 공개, 프로젝트 전체 바인드 마운트, 자동 재시작 |
| `.dockerignore` | 빌드 컨텍스트에서 대용량/불필요 파일 제외 |

## 1. 우분투 서버 준비

Ubuntu 22.04/24.04 LTS 기준. 서버에 SSH로 접속한 뒤:

```bash
# 도커 공식 설치 (Docker Engine + compose 플러그인)
curl -fsSL https://get.docker.com | sudo sh

# 현재 계정에서 sudo 없이 docker 사용 (재로그인 필요)
sudo usermod -aG docker $USER
exit   # 재접속

# 확인
docker --version
docker compose version
```

시간 동기화 확인 (TOTP 2차인증이 서버 시계 기준이므로 중요):

```bash
timedatectl   # "System clock synchronized: yes" 확인. 아니면: sudo timedatectl set-ntp true
```

## 2. 파일 전송 (Windows → 우분투)

**주의: 한글 파일명** 때문에 Windows 탐색기의 "압축(zip)"을 쓰면 안 됩니다(CP949 인코딩으로 압축되어 리눅스에서 파일명이 깨짐). 아래 둘 중 하나를 사용하세요.

**방법 A — scp 직접 전송 (권장, Windows 11 기본 내장):**

PowerShell에서:

```powershell
scp -r "C:\패키지프로그램" 사용자명@서버IP:/home/사용자명/consol-app
```

**방법 B — tar로 묶어 전송:**

```powershell
tar -czf consol-app.tar.gz -C "C:\" "패키지프로그램"
scp consol-app.tar.gz 사용자명@서버IP:/home/사용자명/
# 서버에서: tar -xzf consol-app.tar.gz && mv 패키지프로그램 consol-app
```

전송 후 서버에서 **불필요한 Windows 전용 항목 삭제**:

```bash
cd ~/consol-app
rm -rf python/ venv/ __pycache__/ _slide_preview/ ppt_screenshots/
rm -f setup_and_run.bat server.log _smoke_server.log
```

(`python/`은 Windows용 내장 파이썬, `setup_and_run.bat`은 Windows 실행 스크립트 — 컨테이너에서는 쓰지 않습니다.)

## 3. 경로 마이그레이션 (필수 — 건너뛰면 업로드 목록 전체 소실)

`uploads/_state.json`에 업로드 파일 경로가 Windows 백슬래시(`"uploads\\16ddc914_..."`)로 저장되어 있습니다. `_load_state()`가 시작 시 "실제 파일이 존재하는 항목만" 남기는데, 리눅스에서 백슬래시 경로는 존재하지 않는 파일로 판정되어 **기존 업로드 목록이 전부 제거**됩니다. 서버에서 최초 1회 변환:

```bash
cd ~/consol-app
python3 - <<'EOF'
import json
p = 'uploads/_state.json'
data = json.load(open(p, encoding='utf-8'))
for e in data:
    if 'path' in e:
        e['path'] = e['path'].replace('\\', '/')
json.dump(data, open(p, 'w', encoding='utf-8'), ensure_ascii=False, indent=2)
print(f'{len(data)}건 경로 변환 완료')
EOF
```

(루트의 다른 JSON에는 백슬래시 경로가 없음을 확인했습니다. 이후 운영 중 생성되는 경로는 리눅스 형식으로 저장되므로 이 작업은 최초 1회면 됩니다.)

## 4. 빌드 및 실행

```bash
cd ~/consol-app
docker compose up -d --build
```

확인:

```bash
docker compose ps                      # State: running 확인
docker compose logs -f consol-web      # gunicorn 기동 로그 확인 (Ctrl+C로 빠져나옴)
curl -I http://localhost:5000/         # HTTP 응답 확인
```

사내 PC 브라우저에서 `http://서버IP:5000` 접속 → 로그인 → 업로드 목록/결산기간/계정이 Windows에서 쓰던 그대로 보이는지 확인하세요.

## 5. 일상 운영 명령

모든 명령은 `~/consol-app`에서 실행:

| 작업 | 명령 |
|---|---|
| 상태 확인 | `docker compose ps` |
| 로그 보기 (실시간) | `docker compose logs -f --tail 200` |
| 재시작 | `docker compose restart` |
| 중지 | `docker compose down` |
| 시작 | `docker compose up -d` |
| 의존성(requirements.txt) 변경 시 | `docker compose up -d --build` |

- 서버 재부팅 시 자동 시작됩니다 (`restart: unless-stopped`).
- 중지/재시작 시 60초의 유예시간(`stop_grace_period`)을 두어 업로드 목록(`_state.json`)이 디스크에 flush될 시간을 보장합니다. `docker kill`로 강제 종료하지 마세요 — 마지막 저장이 유실될 수 있습니다.

### 코드 업데이트 절차

이미지에 코드가 들어있지 않으므로 재빌드가 필요 없습니다:

(`requirements.txt`가 바뀐 경우에만 `--build` 필요)

## 6. 백업

DB가 없고 모든 상태가 파일이므로 **디렉터리 전체 백업**이 곧 전체 백업입니다. 핵심은 `uploads/`, `results/`, 루트 `*.json`, `secret_key.bin`.

cron 예시 (매일 새벽 3시, 14일 보관):

```bash
mkdir -p ~/backups
crontab -e
# 아래 줄 추가:
0 3 * * * tar -czf $HOME/backups/consol-$(date +\%Y\%m\%d).tar.gz -C $HOME consol-app && find $HOME/backups -name 'consol-*.tar.gz' -mtime +14 -delete
```

백업 시점에 서비스를 내릴 필요는 없지만, 가능하면 업무시간 외로 잡으세요(업로드 진행 중 스냅샷 방지).

## 7. 보안 / 네트워크

**방화벽 (사내망 전용으로 제한 권장):**

```bash
sudo ufw allow OpenSSH
sudo ufw allow from 사내망대역/24 to any port 5000 proto tcp   # 예: 192.168.10.0/24
sudo ufw enable
```

> 참고: 도커는 자체적으로 iptables 규칙을 추가하므로 ufw만으로 완전히 막히지 않는 경우가 있습니다. 외부 노출이 우려되면 `docker-compose.yml`의 ports를 `"127.0.0.1:5000:5000"`으로 바꾸고 아래 nginx를 앞단에 두는 구성이 확실합니다.

**파일 권한:**

```bash
chmod 600 secret_key.bin auth_config.json distribute_passwords.json
```

**(선택) nginx 리버스 프록시 + HTTPS:**

도메인이 있고 HTTPS가 필요한 경우에만. 앱의 업로드 한도가 50MB(`MAX_CONTENT_LENGTH`)이므로 nginx 한도를 그보다 크게 잡아야 합니다.

```nginx
server {
    listen 80;
    server_name consol.example.com;
    client_max_body_size 60m;          # 앱 한도 50MB보다 크게
    proxy_read_timeout 600s;           # 일괄 검증 등 장시간 요청 대비

    location / {
        proxy_pass http://127.0.0.1:5000;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

HTTPS 인증서는 `sudo apt install certbot python3-certbot-nginx && sudo certbot --nginx` 로 발급.

## 8. 주의사항 정리

1. **워커 수를 늘리지 마세요.** `Dockerfile`의 gunicorn `--workers 1`은 의도된 값입니다(0번 항목의 메모리 캐시 구조). 동시 사용자 증가 시 `--threads` 숫자만 올리세요.
2. **`docker kill` 금지** — `docker compose down`/`restart`를 쓰세요(상태 flush 보장).
3. **`secret_key.bin`을 지우지 마세요** — 지워도 동작은 하지만 전 사용자 세션이 무효화됩니다.
4. **TOTP 로그인 실패가 일괄 발생하면** 서버 시계부터 확인 (`timedatectl`).
5. **시간대**: 컨테이너에 `TZ=Asia/Seoul`이 설정되어 있어 업로드 시각 등 `datetime.now()` 기록이 한국시간으로 남습니다.
6. Windows 원본 PC의 프로그램은 이전 완료 후 **읽기 전용 보관**으로 전환하세요. 양쪽에서 동시에 운영하면 데이터가 갈라집니다.

## 9. 문제 해결

| 증상 | 확인 |
|---|---|
| 컨테이너가 곧바로 죽음 | `docker compose logs` — import 오류면 파일 누락(전송 단계 재확인) |
| 업로드 목록이 비어 보임 | 3단계 경로 마이그레이션 수행 여부, `uploads/` 디렉터리 전송 여부 |
| 한글 파일명 깨짐 | zip으로 전송했는지 확인 → scp/tar로 재전송 |
| 50MB 초과 업로드 실패 | 앱 한도(`app.py`의 `MAX_CONTENT_LENGTH`)이며 정상 동작. nginx 사용 시 `client_max_body_size`도 확인 |
| 일괄 검증 중 504/timeout | gunicorn `--timeout 600` 및 nginx `proxy_read_timeout` 확인 |
