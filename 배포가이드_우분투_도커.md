### 코드 vs 데이터 구분 (CI/CD의 핵심 — 코드로 전수 확인한 사실)

이 앱은 DB 없이 모든 상태를 프로젝트 루트의 파일로 관리합니다. **git이 추적하는 것은 코드뿐**이고, 서버에서 변하는 데이터는 Jenkins 배포가 절대 건드리지 않습니다.

| 구분 | 항목 | 배포 방법 |
|---|---|---|
| **코드 (git 추적)** | `*.py`, `templates/`, `requirements.txt`, `Dockerfile`, `docker-compose.yml`, `Jenkinsfile`, `consol_template.json`(읽기전용 확인됨) | git push → Jenkins 자동 배포 |
| **데이터 (git 제외, 운영서버에만 존재)** | `uploads/`(패키지·`_state.json`·journals), `results/`, `secret_key.bin`, `auth_config.json`, `permission_groups.json`, `years_config.json`, `wce_overrides.json`, `fx_rates.json`, `cash_worksheet_manuals.json`, `prior_consolidated.json`, `consol_groups.json`, `consol_journals.json`, `distribute_passwords.json`, `cf_mapping_v2_draft.json`, `distribute_template/*.xlsm`(관리자 UI로 업로드됨) | 최초 1회 전송 후 운영서버에서만 관리, 백업으로 보호 |
| **데이터 (읽기전용, 수동 교체)** | `필수제출회사리스트.xlsx` | 변경 시 운영서버에 scp로 직접 교체 |

위 데이터 목록은 `app.py`/`consol_schema.py`/`distribute_builder.py`의 저장 함수(`_save_credentials`, `save_journals`, `save_template` 등)를 전부 추적해 확인한 것입니다. `.gitignore`에 이미 반영되어 있습니다.

### 운영 구조상 주의 2가지 (왜 이렇게 설계했는가)

1. **gunicorn 워커는 반드시 1개** — 업로드 목록이 프로세스 메모리 전역변수에 캐시되고 백그라운드 스레드가 `uploads/_state.json`으로 비동기 저장하는 구조(`app.py:7203`). 워커 2개 이상이면 워커 간 목록이 어긋남. 동시 사용자는 스레드 수(`--threads`)로 감당.
2. **이미지에 코드를 굽지 않고 디렉터리 전체를 바인드 마운트** — 코드와 데이터가 한 디렉터리에 섞여 있는 현재 구조에서 가장 안전. 배포 = "추적 파일만 rsync + 컨테이너 재시작"이라 이미지 재빌드는 `requirements.txt` 변경 시에만 발생.

### 3.2 경로 마이그레이션 (필수 — 건너뛰면 기존 업로드 목록 전체 소실)

`uploads/_state.json`에 업로드 경로가 Windows 백슬래시(`"uploads\\16ddc914_..."`)로 저장되어 있습니다. 시작 시 `_load_state()`가 "실존 파일만" 남기는데 리눅스에서 백슬래시 경로는 없는 파일로 판정되어 목록이 전부 제거됩니다. 최초 1회 변환:

**[운영서버 193]**

```bash
cd /opt/consol-app
sudo python3 - <<'EOF'
import json
p = 'uploads/_state.json'
data = json.load(open(p, encoding='utf-8'))
for e in data:
    if 'path' in e:
        e['path'] = e['path'].replace('\\', '/')
json.dump(data, open(p, 'w', encoding='utf-8'), ensure_ascii=False, indent=2)
print(f'{len(data)}건 경로 변환 완료')
EOF
```

(루트의 다른 JSON에는 백슬래시 경로가 없음을 확인했습니다. 이후 생성되는 경로는 리눅스 형식이므로 1회면 충분합니다.)

## 5. nginx + 도메인 + SSL

### 5.1 DNS 확인

**[운영서버 193]**

```bash
dig +short consol.sae-a.com    # 운영서버(193) IP가 나와야 함
```

안 나오면 DNS 관리자에게 A 레코드 등록을 요청합니다.

### 5.2 인증서 파일 확인

`/etc/nginx/ssl/`의 실제 파일명을 확인하고 아래 설정의 경로를 맞춥니다.

**[운영서버 193]**

```bash
ls -l /etc/nginx/ssl/

# 인증서가 consol.sae-a.com을 커버하는지 + 만료일 확인 (파일명은 실제에 맞게)
openssl x509 -in /etc/nginx/ssl/consol.sae-a.com.crt -noout -subject -ext subjectAltName -enddate
```

- `subjectAltName`에 `consol.sae-a.com` 또는 `*.sae-a.com`이 있어야 합니다.
- **crt 파일은 중간 인증서가 포함된 fullchain이어야** 브라우저 경고가 없습니다. 확인: `openssl crl2pkcs7 -nocrl -certfile 인증서.crt | openssl pkcs7 -print_certs -noout` 결과에 인증서가 2개 이상 나오면 체인 포함. 1개뿐이면 IT부서에서 중간 인증서(chain/CA bundle)를 받아 이어붙이세요: `cat 도메인.crt 중간CA.crt > fullchain.crt`
- 개인키 권한: `sudo chmod 600 /etc/nginx/ssl/*.key`

### 5.3 nginx 설치 및 설정

**[운영서버 193]**

```bash
sudo apt -y install nginx
```

`/etc/nginx/sites-available/consol` 생성 (인증서 파일명 2곳은 실제에 맞게 수정):

```nginx
# HTTP → HTTPS 리다이렉트
server {
    listen 80;
    server_name consol.sae-a.com;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    server_name consol.sae-a.com;

    ssl_certificate     /etc/nginx/ssl/consol.sae-a.com.crt;   # fullchain (체인 포함)
    ssl_certificate_key /etc/nginx/ssl/consol.sae-a.com.key;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers on;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1d;

    # 앱 업로드 한도가 50MB(MAX_CONTENT_LENGTH)이므로 그보다 크게
    client_max_body_size 60m;
    # 패키지 일괄 검증 등 장시간 요청 대비 (gunicorn timeout 600과 맞춤)
    proxy_read_timeout 600s;
    proxy_send_timeout 600s;

    location / {
        proxy_pass http://127.0.0.1:5000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

활성화 및 확인:

```bash
sudo ln -s /etc/nginx/sites-available/consol /etc/nginx/sites-enabled/
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t && sudo systemctl reload nginx

curl -I https://consol.sae-a.com/          # 200 또는 302
curl -I http://consol.sae-a.com/           # 301 (https 리다이렉트)
```

**[웹브라우저]** `https://consol.sae-a.com` 접속 → 자물쇠 아이콘(인증서 유효) → 로그인 → 업로드 목록/결산기간/계정이 Windows에서 쓰던 그대로인지 확인하세요.

### 5.4 인증서 갱신 (운영 메모)

`/etc/nginx/ssl/` 인증서는 자동 갱신이 없습니다. **만료일을 달력에 등록**해 두고, IT부서에서 새 인증서를 받으면:

**[운영서버 193]**

```bash
sudo cp 새인증서.crt /etc/nginx/ssl/consol.sae-a.com.crt    # fullchain인지 5.2처럼 확인
sudo cp 새키.key     /etc/nginx/ssl/consol.sae-a.com.key
sudo chmod 600 /etc/nginx/ssl/consol.sae-a.com.key
sudo nginx -t && sudo systemctl reload nginx                 # 무중단 적용
```

만료 30일 전 알림 (선택):

```bash
# sudo crontab -e 에 추가 — 만료 30일 이내면 매일 syslog에 경고
0 9 * * * openssl x509 -in /etc/nginx/ssl/consol.sae-a.com.crt -noout -checkend 2592000 || logger -t ssl-expiry "consol.sae-a.com 인증서 만료 30일 이내!"
```

## 6. Jenkins CI/CD (젠킨스서버 176 → 운영서버 193 원격 배포)

### 구조 요약

Jenkins는 **젠킨스서버(176)에서** 돌면서, git에서 코드를 받아 문법검사 후 **SSH로 운영서버(193)에** 추적 파일만 rsync하고 원격으로 `docker compose` 재시작을 실행합니다. 이를 위한 연결고리는 2개입니다:

1. **176의 jenkins 계정 → GitHub** : Deploy Key (코드 받기)
2. **176의 jenkins 계정 → 193의 saea 계정** : SSH 키 (배포하기)

하나의 키 쌍을 만들어 두 곳에 모두 등록합니다.

### 6.1 젠킨스서버 준비 + SSH 키 생성

**[젠킨스서버 176]**

```bash
# 파이프라인이 사용하는 도구 확인
sudo apt -y install git rsync python3

# jenkins 계정의 SSH 키 생성 (.ssh 디렉터리가 없을 수 있으므로 먼저 생성)
sudo -u jenkins mkdir -p /var/lib/jenkins/.ssh
sudo chmod 700 /var/lib/jenkins/.ssh
sudo chown jenkins:jenkins /var/lib/jenkins/.ssh
sudo -u jenkins ssh-keygen -t ed25519 -f /var/lib/jenkins/.ssh/id_ed25519 -N ""

# 공개키 출력 — 이 내용을 복사해 둠 (6.2와 6.3에서 사용)
sudo -u jenkins cat /var/lib/jenkins/.ssh/id_ed25519.pub
```

> `jenkins`는 Jenkins 설치 시 자동 생성된 시스템 계정 이름 그대로입니다. 본인 계정명으로 바꾸지 마세요. `sudo: unknown user jenkins`가 나오면 이 서버에 Jenkins가 설치되지 않은 것입니다 (`id jenkins`로 확인).

### 6.2 운영서버에 배포 키 등록

6.1에서 복사한 **공개키(.pub)** 를 운영서버 배포 계정의 authorized_keys에 추가합니다.

**[운영서버 193]**

```bash
mkdir -p ~/.ssh && chmod 700 ~/.ssh
echo '여기에_6.1의_공개키_한줄_붙여넣기' >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys
```

**[젠킨스서버 176]** — 접속 테스트 (최초 1회 호스트 키 등록, yes 입력):

```bash
sudo -u jenkins ssh saea@193서버IP "docker ps && ls /opt/consol-app/app.py"
# 비밀번호 없이 접속되고, docker ps와 파일 목록이 나오면 성공
```

비밀번호를 물으면 6.2의 공개키 등록이 잘못된 것이고, `permission denied`로 docker가 실패하면 3.3의 `usermod -aG docker saea` 누락입니다.

### 6.3 GitHub 접근 설정

**[웹브라우저]** GitHub 저장소 → Settings → **Deploy keys** → Add deploy key → 6.1의 **공개키(.pub)** 붙여넣기 (read-only면 충분)

**[젠킨스서버 176]** — GitHub 호스트 키 등록 (최초 1회, yes 입력):

```bash
sudo -u jenkins ssh -T git@github.com
# "You've successfully authenticated, but GitHub does not provide shell access." 면 정상
```

**[웹브라우저]** Jenkins UI(`http://176서버IP:8080`) → Manage Jenkins → Credentials → Add:
- Kind: **SSH Username with private key**, Username: `git`
- Private Key → Enter directly: **개인키** 내용 붙여넣기 (젠킨스서버에서 `sudo -u jenkins cat /var/lib/jenkins/.ssh/id_ed25519` 출력 — `.pub` 없는 쪽)

(사내 GitLab도 동일 — Deploy Key 또는 Access Token 방식)


### 6.5 파이프라인 Job 생성

**[웹브라우저]** Jenkins UI에서:

1. **New Item** → 이름 `consol-app-deploy` → **Pipeline** 선택
2. Pipeline 섹션:
   - Definition: **Pipeline script from SCM**
   - SCM: **Git**, Repository URL: `git@github.com:회사조직/consol-app.git`

### 6.6 자동 트리거

`Jenkinsfile`에 `pollSCM('H/2 * * * *')`이 들어 있어 **2분 간격으로 git 변경을 감지해 자동 배포**합니다. 별도 설정 없이 동작합니다 (단, Job을 최초 1회는 수동 빌드해야 트리거가 등록됩니다).

젠킨스서버가 인터넷에서 접근 가능하고 즉시 배포를 원하면 webhook으로 전환할 수 있습니다: GitHub 저장소 → Settings → Webhooks → `http://176서버주소:8080/github-webhook/` 등록 + Jenkinsfile에서 `triggers` 블록 제거 + Job 설정에서 "GitHub hook trigger" 체크. 사내망 전용이라면 pollSCM을 그대로 쓰는 것이 간단합니다.

## 8. 보안

### 8.1 방화벽 (ufw)

**[운영서버 193]**

```bash
sudo ufw allow OpenSSH                # 176의 Jenkins와 관리자 SSH 접속 경로
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
sudo ufw status
```

**[젠킨스서버 176]**

```bash
sudo ufw allow OpenSSH
sudo ufw allow from 사내망대역/24 to any port 8080 proto tcp   # Jenkins UI는 사내망만 (예: 192.168.10.0/24)
sudo ufw enable
```

- 앱 포트 5000은 `127.0.0.1` 바인딩이라 방화벽과 무관하게 외부 접근이 불가합니다.
- 참고: 도커가 직접 publish하는 포트는 ufw를 우회할 수 있으나, 이 구성은 localhost 바인딩이므로 해당 없음. **운영서버에서 기존 운영 중인 다른 컨테이너**가 `0.0.0.0`으로 포트를 열고 있다면 그쪽은 별도 점검 필요.

### 8.2 파일 권한 (3.3에서 수행했는지 확인)

**[운영서버 193]**

```bash
ls -l /opt/consol-app/secret_key.bin /opt/consol-app/auth_config.json /opt/consol-app/distribute_passwords.json
# -rw------- (600) 이어야 함
sudo ls -l /etc/nginx/ssl/    # 개인키(.key)도 600 확인
```

### 8.3 기타 권장

- 두 서버 모두 SSH 비밀번호 로그인 비활성화(키 인증만): `/etc/ssh/sshd_config`에서 `PasswordAuthentication no` — 단, **6.2의 키 등록과 접속 테스트가 끝난 뒤에** 적용할 것
- Jenkins 관리자 외 계정 생성 금지, "Anyone can do anything" 권한 설정 금지 (기본값 유지)
- `sudo apt -y install fail2ban` — SSH 무차별 대입 차단 (기본 설정으로 충분)
- git 저장소는 반드시 **private** — 코드 자체에 내부 계정 구조·검증 로직이 들어 있음

## 9. 문제 해결

| 증상 | 확인/조치 |
|---|---|
| 컨테이너가 곧바로 죽음 | [193] `docker compose logs` — import 오류면 파일 누락. Jenkins 배포 직후라면 해당 빌드의 Syntax Check 통과 여부 확인 |
| 5000 포트 충돌로 기동 실패 | [193] `docker ps`로 점유 컨테이너 확인 — compose 호스트 포트·nginx `proxy_pass`·Jenkinsfile 헬스체크 포트를 함께 변경 |
| 업로드 목록이 비어 보임 | 3.2 경로 마이그레이션 수행 여부, `uploads/` 전송 여부 |
| 한글 파일명 깨짐 | zip으로 전송한 경우 → scp로 재전송 |
| 브라우저 인증서 경고 | ① 인증서 SAN에 `consol.sae-a.com` 포함 여부 ② crt가 fullchain(중간 인증서 포함)인지 — 5.2 참조 |
| 인증서 만료 | IT부서에서 새 인증서 수령 → 5.4 절차로 교체 (자동 갱신 없음) |
| 50MB 초과 업로드 실패 | 앱 자체 한도(정상). nginx `client_max_body_size 60m`이 443 블록에 있는지 확인 |
| 일괄 검증 중 504 | nginx `proxy_read_timeout 600s` + gunicorn `--timeout 600` 확인 |
| TOTP 로그인 일괄 실패 | [193] 서버 시계 (`timedatectl`) |
| Jenkins: `Host key verification failed` | [176] `sudo -u jenkins ssh saea@193서버IP` 와 `sudo -u jenkins ssh -T git@github.com` 를 1회씩 실행해 yes로 호스트 키 등록 (6.2, 6.3) |
| Jenkins: rsync/ssh `Permission denied` | ① 6.2 공개키가 193의 `~/.ssh/authorized_keys`에 등록됐는지 ② [193] `/opt/consol-app` 소유자가 배포 계정인지 (`sudo chown -R saea:saea /opt/consol-app`) |
| Jenkins: 원격 docker 명령 실패 | [193] 배포 계정이 docker 그룹인지 (`groups saea`), 아니면 `sudo usermod -aG docker saea` 후 재접속 |
| Jenkins가 push를 감지 안 함 | Job을 최초 1회 수동 빌드했는지(pollSCM 등록 조건), Git 자격증명 유효성 |
| push했는데 코드가 반영 안 됨 | 해당 파일이 `.gitignore`에 걸려 추적 제외됐는지 `git status`로 확인 |
